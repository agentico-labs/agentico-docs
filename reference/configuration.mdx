A single reference for every knob in effy.

## Environment variables

| Variable | Required | Default | Purpose |
| --- | --- | --- | --- |
| `EFFY_JWT_SECRET` | production | — | HMAC secret used to verify (and sign) session tokens. |
| `EFFY_JWT_ISSUER` | no | `effy` | Expected `iss` claim. |
| `EFFY_JWT_AUDIENCE` | no | `effy-agent` | Expected `aud` claim. |
| `EFFY_DEV_AUTH` | dev only | unset | `1` enables `x-effy-*` dev headers. Must be unset in production. |
| `SUPERMEMORY_API_KEY` | yes | — | Supermemory key backing memory, personas, and tools. |

## Scripts

```bash
npm run dev         # eve dev REPL (set EFFY_DEV_AUTH=1 for local auth)
npm run typecheck   # tsc
npm test            # node --test (auth self-check)
npm run build       # eve build
npm run start       # eve start
```

## The model

```ts title="agent/agent.ts"
import { defineAgent } from "eve";

export default defineAgent({
  model: "anthropic/claude-sonnet-4.6",
});
```

Change the model by editing this one line. It routes through the Vercel AI Gateway; see [Deployment](/guides/deployment#the-model).

## Validation limits and caps

effy validates every input at its trust boundary. These are the bounds it enforces:

### Memory

| Field | Limit |
| --- | --- |
| `key` | 1–80 chars, `^[a-z0-9_.-]+$` |
| `value` | 1–4,000 chars |
| Memories injected per turn | 50 (most recent first) |

### Persona / profile

| Field | Limit |
| --- | --- |
| `name` | 1–120 chars |
| `personality` | 1–4,000 chars |
| `instructions` | 1–8,000 chars |

The profile schema is `.strict()` — unknown fields are rejected.

### Personal tools

| Field | Limit |
| --- | --- |
| Tool `name` (input) | `^[a-z][a-z0-9_]{0,38}$`, stored as `u_<name>` |
| `description` | 1–500 chars |
| `code` | 1–20,000 chars |
| `runtime` | `bash` \| `python` \| `node` |
| `params` | up to 10 |
| Param `name` | `^[a-z][a-z0-9_]{0,30}$` |
| Param `description` | up to 200 chars |
| Tool output returned to the model | clipped at 10,000 chars |

### Auth

| Setting | Value |
| --- | --- |
| JWT algorithm | HS256 |
| Clock skew tolerance | 30 seconds |

## Sandbox network policy

Egress starts at `deny-all` because the sandbox can run user-authored code. The backend is chosen so domain allow-lists actually work: **microsandbox** locally (Docker can't scope egress by domain), **Vercel Sandbox** in production.

```ts title="agent/sandbox/sandbox.ts"
const backend = process.env.VERCEL
  ? defaultBackend({ vercel: { networkPolicy: "deny-all" } })
  : microsandbox({ networkPolicy: "deny-all" });
```

Per session, effy opens egress only to the domains the caller has approved via the `allow_domain` tool — a per-`(tenant, user)` allow-list applied with `sandbox.setNetworkPolicy(...)`. Domain allow-lists are enforced on the Vercel and microsandbox backends; Docker is all-or-nothing and just-bash has no network. See [Per-user tools](/concepts/custom-tools#the-sandbox).

## Approval-gated tools

Three tools require human approval before they run. This is product policy expressed with eve's approval flow — adjust it by changing each tool's `approval` option:

- `forget` — deletes a memory
- `create_tool` — persists executable code
- `delete_tool` — removes a personal tool
