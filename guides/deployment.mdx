effy deploys like any eve agent. This page covers the variables it reads, how the model is configured, and what you must change before handling real multi-tenant data.

## Environment variables

| Variable | Required | Purpose |
| --- | --- | --- |
| `EFFY_JWT_SECRET` | production | HMAC secret used to verify session tokens. |
| `EFFY_JWT_ISSUER` | no (`effy`) | Expected `iss` claim. |
| `EFFY_JWT_AUDIENCE` | no (`effy-agent`) | Expected `aud` claim. |
| `EFFY_DEV_AUTH` | dev only | `1` enables `x-effy-*` dev headers. **Must be unset in production.** |
| `SUPERMEMORY_API_KEY` | yes | Supermemory key backing memory, personas, and tools. |

The model also needs gateway credentials — see [the model](#the-model) below.

## The model

The base model is configured in `agent/agent.ts`:

```ts
import { defineAgent } from "eve";

export default defineAgent({
  model: "anthropic/claude-sonnet-4.6",
});
```

The model routes through the Vercel AI Gateway. Provide its credentials per the [eve deployment docs](https://eve.dev/docs). To change models, edit this one line — the rest of effy is model-agnostic.

## Build and run

```bash
npm run build   # eve build
npm run start   # eve start
```

## Production checklist

<Steps>
  <Step title="Set a strong JWT secret">
    Set `EFFY_JWT_SECRET` to a high-entropy secret from your secret manager. Without it, token auth is not configured and bearer requests are rejected.
  </Step>
  <Step title="Unset dev auth">
    Ensure `EFFY_DEV_AUTH` is **not** set. It lets any caller claim any identity and persona.
  </Step>
  <Step title="Set the Supermemory key">
    Memory, personas, and tools persist in Supermemory. Set `SUPERMEMORY_API_KEY` and keep it server-side. See [below](#supermemory).
  </Step>
  <Step title="Keep the sandbox locked down">
    Egress is `deny-all`; hosts open only when a user approves them via `allow_domain`. Keep `BASE_ALLOWED_EGRESS` minimal. See [Per-user tools](/concepts/custom-tools#the-sandbox).
  </Step>
  <Step title="Enforce session ownership">
    Route auth identifies the caller but does not decide which sessions it may access. Enforce that ACL in your app if tenants share routes. See [Multi-tenancy](/concepts/multi-tenancy#session-ownership-is-yours-to-enforce).
  </Step>
  <Step title="Bound user-tool risk">
    Keep `create_tool` gated on approval, and cap tool count, code size, and runtime. Treat all user code as untrusted and idempotent.
  </Step>
</Steps>

## Supermemory

Memory, personas, and tools all persist in [Supermemory](https://supermemory.ai) through one client (`lib/supermemory.ts`). Set the key:

```bash
SUPERMEMORY_API_KEY=sk-...   # required; keep it server-side
```

Each store is reached only through a small interface (`MemoryStore`, `ProfileStore`, `ToolStore`), so the backend stays swappable without touching the rest of the agent:

```ts
// agent/lib/memory-store.ts — the interface the agent depends on
export interface MemoryStore {
  list(scope: TenantScope, options: { limit: number }): Promise<Memory[]>;
  put(scope: TenantScope, memory: { key: string; value: string }): Promise<Memory>;
  delete(scope: TenantScope, key: string): Promise<boolean>;
}
```

How effy keeps it multi-tenant-safe:

- **Isolation is by container tag.** Each `(tenant, user)` and namespace (memory, persona, tool) maps to its own Supermemory container tag, and every read is scoped to the caller's tag.
- **Identity is by key within a tag**, so overwrite and delete never depend on a globally unique id.
- Reads **degrade gracefully** if Supermemory is unreachable; writes surface the error to the calling tool.

<Note>
  Without `SUPERMEMORY_API_KEY`, the stores throw on use and the resolvers add no memory, persona, or tools. In local dev, effy auto-loads it from `.env.local` or `.env`; in production the platform injects it.
</Note>
